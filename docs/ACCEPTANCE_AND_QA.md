# VibeReader Standalone Dev 验收方法与验收矩阵

最后更新：2026-05-23

## 1. 基线验收

### 1.1 当前已验证

命令：

```bash
npm run build
```

结果：

- 通过。
- Vite 编译成功。
- 存在单 chunk 大于 500 kB 的体积警告。
- 主 bundle 约 2.27 MiB。

这些警告不是当前阻塞项，但后续需要通过 code splitting 处理。

### 1.2 每次改动后的最低验证

```bash
npm run build
```

如果已经完成 Tauri 迁移，还需要：

```bash
npm run tauri:dev
```

若做了打包相关修改，再运行：

```bash
npm run tauri:build
```

## 3. 手工验收流程

### 3.1 PDF 核心闭环

1. 启动 Tauri 桌面应用。
2. 点击“打开文件”。
3. 选择 demo PDF。
4. 等待解析进度结束。
5. 在阅读器中翻到第二页。
6. 放大到 125%。
7. 选择一段文字。
8. 点击“注入上下文”。
9. 在 AI 面板中追加问题：“用中文解释这段话，并指出它在全文中的作用。”
10. 等待 AI 回复。

通过标准：

- 阅读器没有空白。
- 选区文本进入 Chat。
- AI 回复不丢失上下文。
- 会话刷新后仍可看到历史消息。

### 3.2 通用阅读闭环

1. 打开 Markdown 文件。
2. 确认标题、列表、代码块可读。
3. 选中一段正文。
4. 注入 AI。
5. 让 AI 生成 3 张闪卡。

通过标准：

- 非 PDF 文档能进入同一阅读工作台。
- AI 能围绕选区内容回答。
- 闪卡面板能显示结果或给出明确生成入口。

### 3.3 失败状态验收

1. 打开一个非支持格式文件。
2. 打开一个损坏 PDF。
3. 清空 API key 后发送问题。
4. 网络断开时打开 PDF。

通过标准：

- 所有错误都有用户可读提示。
- 控制台不能出现持续刷屏错误。
- UI 不进入永久 loading。

## 4. 自动化测试建议

当前项目没有测试框架。建议先补最小测试，而不是一次性引入复杂 e2e。

### 4.1 单元测试优先对象

- `aiService` SSE 解析。
- `storage` 会话标题提取和 IndexedDB 封装。
- `documentService` 文件类型识别。
- `vibeParser` 结构化解析。

### 4.2 推荐工具

- Vitest：配合 Vite 迁移。
- Playwright：只做 2-3 条关键 e2e。

### 4.3 最小 e2e

1. 启动应用。
2. 打开 demo Markdown。
3. 看到正文。
4. 打开 demo PDF。
5. 看到 PDF 页码。

## 6. 发布前阻塞项

以下任一项失败，都不能说 Hackathon demo ready：

- Tauri 应用无法启动。
- 不能从系统文件选择器打开 PDF。
- PDF 阅读器空白或不能翻页。
- 选中文本不能注入 AI。
- 无 API key 时直接崩溃。
- 演示资产缺失。
- 演示脚本不能在 3 分钟内跑通。
