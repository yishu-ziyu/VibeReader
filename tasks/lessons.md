# 项目工作方法修正

## 2026-06-02：产品推进先 BDD/PM，技术实现再工程化

开发 VibeReader 时，默认先站在产品经理和 BDD 视角推进：

- 先定义用户场景、业务规则、Given / When / Then、验收标准和边界条件；
- 先判断功能是否加强 source-grounded reading、local-first、artifact 化输出和可验证阅读闭环；
- 只有进入具体技术实现时，才切换到工程师视角处理模块边界、测试、数据结构、性能和构建验证；
- 不要一开始就从技术可实现性或代码结构出发替代产品行为定义。

## 2026-06-02：少做阶段命名，多推进功能闭环

和用户讨论产品进展时，不要用 “MVP+” 这类阶段标签占篇幅。用户关心的是具体功能闭环是否完成。

本项目的产品方向应持续对齐：参考 Vibero/Vibro 的论文阅读体验，但从 Zotero 绑定形态扩展为通用本地阅读系统。推进时直接围绕未完成闭环工作：

- 选中原文；
- 生成解释 / 闪卡 / 笔记等卡片；
- 显示来源 span 或 inference 标记；
- 保存为 artifact；
- 能回到原文位置。

## 2026-06-02：Skim Map 和 Lens Card 是不同产品区域

不要把 Skim Map 和 Lens Card 都塞进右侧栏。

- Skim Map 属于文档全局结构和阅读路径，应放在 PDF 正文左侧；
- Lens Card 属于局部原文旁注和阅读资产，应放在 PDF 正文右侧 Notes；
- 右侧用户文案用 Notes / 旁注，不把工程词 Artifacts 暴露给用户；
- Skim Map 第一版不能默认调用 LLM，生成必须由用户显式触发。

## 2026-06-02：Skim Map 不是独立左面板，而是阅读纸面的边注

不要把 Skim Map 和 Reader 做成两张并排卡片。用户要的是读报纸/读文章时，正文旁边自然长出结构边注的感觉。

- Skim Map 和 Reader 应共享同一个 reading surface；
- Skim Map 是这张阅读纸面的左边注，不是外部工具栏；
- Reader 是同一张纸面的正文区域；
- 视觉上可以保留弱分隔，但不要用两个独立阴影、两个独立圆角、两个独立卡片间距制造割裂感。

## 2026-06-04：模型服务配置是 AI Agent 产品主流程

大模型服务配置不是 Chat 的附属设置，而是所有 agentic reading 能力的前置条件。

- 配置入口必须在 Notes、Lens Card 等非聊天工作流里也可见；
- 缺 key、坏 key、代理/CORS、provider 不可用时必须给可理解提示；
- 不让用户把模型配置失败误判为阅读器、PDF 或 Notes 功能坏了；
- 真实 API key 只能存在本机 localStorage 或安全配置里，不写入 git、文档、日志或记忆。
